# persona-builder

대표/사업가의 **암묵적 판단·반응 패턴**을 PERSONA.md 한 장으로 외부화하는 스킬. BUSINESS.md(=무엇을: 서비스·가격·규칙)와 짝을 이루는 **행동 사양서(=어떻게/누가: 판단·반응·말투)**를 만든다.

> 핵심 원칙: 문서에 *이미 적힌 것*을 요약하지 말고, **규칙 뒤의 암묵지**(불확실할 때의 결정, 감정 트리거, 레드라인, 직감, 말버릇)를 끌어낸다. 페르소나의 진짜 결은 규칙이 아니라 **예외와 충돌**에서 나온다.

## 4단계 워크플로우

### Phase 1 — 단계적 인터뷰 (먼저, 반드시)

데이터 분석 **전에** 인터뷰로 맥락을 쌓는다. **한 번에 여러 질문을 던지되, 라운드를 거듭하며 점점 깊이 판다.** 보통 3라운드.

먼저 "이미 문서로 아는 것"을 한 번 정리해 보여주고 _중복 질문을 피한다._

- **R1 — 판단의 골격**: ① 불확실할 때 지르나/미루나 + 최근 사례 ② 두 선택지가 팽팽할 때 기우는 _단 하나의 기준_ ③ "내가 틀렸었다"고 바꾼 결정 ④ 돈 줘도 안 받을 고객/일 ⑤ 상담 중 "이건 아닌데"/"되겠다" 신호 ⑥ 가장 에너지 솟는 일/하기 싫은 일 ⑦ 자주 쓰는 말투(자각하는 것) ⑧ 페르소나가 절대 하면 안 되는 것
- **R2 — 경계와 모순**: R1 답에서 *애매하게 남은 지점*을 후벼판다. 거절 사례 해부(결정적 순간), "충분하다"는 직관의 정체, 솔직함·책임의 비용(손해 봐도 솔직했던 사례), 신뢰·약속을 행동으로 검증하는 법, 실패 책임 귀속. + **비즈니스 페르소나 브릿지**: "사업이 사람이라면? 본인과 다른 점?"
- **R3 — 목소리·메타 보정**: 상담 여는/닫는 말, 말버릇·비유·외부예시, 말 vs 글 톤 차이. + **메타 질문(중요)**: "개인 페르소나 = 있는 그대로(약점 포함 거울) / 비즈니스 페르소나 = 약점 보정한 이상형" 구분이 맞나? 흔들릴 때 겉으로 드러나는 신호는?

> 말버릇은 본인이 자각 못 하는 경우가 많다 → "그건 녹취/기록에서 뽑아드리겠다"로 넘기고 Phase 2에서 채운다.

### Phase 2 — 다중 소스 흡수 (병렬 서브에이전트)

가용 소스를 자동 탐지해 **각각 distill 서브에이전트**로 돌린다. 원본 덤프 금지, distilled signal만.

가능한 소스:

- **통화 녹취 (PLAUD)** — 가장 풍부. CLI: `npm i -g @plaud-ai/cli` → 사용자가 `plaud login`(브라우저 OAuth, 직접) → `plaud files -s 100`로 ID 열거(32-hex 정규식 추출) → `plaud transcript <id> -o <file>`. **PII 포함 → repo 아닌 임시 디렉토리(job tmp 등)에만 저장.** 튜토리얼/샘플 녹취는 제외.
- **repo 문서** — clients/_/notes·analysis, plan/, ideas/, admin/, .claude/memory/_.jsonl(과거 결정·실패가 금광).
- **Claude Code 세션** — claude-mem 검색 도구(있으면)로 과거 대화의 결정·반응 흔적.
- **Notion** — 개인/회사 워크스페이스. notion MCP(직접 API)로 검색·정독. 회고·원칙·환불·상담·생각 키워드 표적 탐색.

각 서브에이전트에 **6차원 증거 스키마**를 준다 (항목 = 신호 한 줄 + 짧은 직접 인용 + 출처):

1. 의사결정 패턴 2. 반응 패턴 3. 레드라인·거절 4. 가치관·철학 5. 목소리·말버릇 6. 자기인식

- Phase 1에서 확정한 기준점을 함께 줘서 **확증/반박/⭐새발견**으로 표시하게 한다.

### Phase 3 — 합성 → PERSONA.md

교차검증(소스 간 수렴 확인) 후 한 장으로 합성. 실명은 **익명화**(repo 동기화·공개 대비). 구조:

```
# {사업명} — Persona
> 목적 / BUSINESS.md와의 관계 / 구성 / 출처·작성일·익명화 고지

## PART A — {대표} 판단 엔진 (충실한 거울)
A1 의사결정 메커니즘 / A2 반응 패턴 / A3 레드라인 / A4 가치관 / A5 약점(그대로 기록)

## PART B — {사업} 비즈니스 페르소나 (운영용 이상형)
B1 성격(A 대비 보정표) / B2 핵심 행동 원칙 / B3 상담 진행 패턴
   ※ 메타답(거울 vs 이상형)에 따라 운영 체화 대상 결정. 보통 PART B.

## 목소리 가이드 (말/글 분리, 시그니처 표현, 금지어)
## 판단 휴리스틱 (IF-THEN — AI가 직접 적용)
## 시나리오 few-shot (상황별 반응 예시 2~3개)
```

각 항목은 **짧은 실제 인용**으로 뒷받침해 "진짜 그 사람"처럼 느껴지게 한다. A5(약점)를 빼면 거울이 가짜가 된다 — 반드시 포함하되, B에서 보정 방향을 명시.

### Phase 4 — 세션 자동 적용

PERSONA.md를 매 세션 로드되게 CLAUDE.md에 import 한 줄을 추가한다:

```markdown
## Persona (항상 적용)

고객 상담·판단을 흉내 낼 때 PERSONA.md의 PART B를 체화한다.
@PERSONA.md
```

(대안: SessionStart 훅으로 주입. 하지만 `@import`가 가장 단순하고 native.)

## 주의

- **PII**: 통화 녹취·노션엔 실명/회사명. 원본은 repo 밖 임시 저장, 최종 PERSONA.md엔 익명화([고객A] 등).
- **과한 일반화 금지**: 인용 1~2개로 단정하지 말고, 여러 소스가 수렴할 때 패턴으로 승격. 모순되는 신호(예: "느리다" 자기서사 vs 실제 추진력)는 _모순 그대로_ 기록하는 게 더 정확하다.
- **소프트 신호 발굴**: 명시적 발화만 보지 말고 행동 패턴(예: 면전 거절 대신 "검토하겠다"로 드롭)을 잡는다.
- 완성 후 사용자에게 **사실과 다른 결을 교정**받는 라운드를 반드시 한 번 둔다.
